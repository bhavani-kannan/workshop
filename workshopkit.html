5-Day Manufacturing Decision Workshop

A structured five-day engagement to surface the decisions that drive plant performance, validate data readiness, define decision logic, and arrive at a build-ready use case.

What the client walks away with

Each day produces a concrete output: a decision backlog, a selected use case, a data readiness assessment, a decision model, and a validated implementation path. No slide decks that sit on shelves.

Day 1
3 sessions
Identify Critical Manufacturing Decisions
Identify what gets decided, how, and where the process consistently breaks down.
Operational friction map
Decision inventory (15-25 decisions)
Priority shortlist (5-7 decisions)
Day 2
3 sessions
Select the Priority Use Case
Choose one decision to take forward, based on impact, frequency, and data access.
Scored use case list
Selected use case
Selection rationale

Objective of the day

The shortlist from Day 1 gets tested against practical criteria. Each decision is scored on operational impact, decision frequency, and the likelihood that the required data is available and accessible. By end of day, one use case is selected to carry forward.

Session 1
Use Case Scoring Score each shortlisted decision against a structured criteria set covering impact, frequency, and data access.
Session 2
Stakeholder Validation Review the top two or three candidates with operations and IT leads. Surface concerns about feasibility or priority.
Session 3
Use Case Selection Agree on one use case and document what the agent is expected to decide, how often, and for whom.

End of Day 2 outputs

Scored use case matrix
Selected use case with rationale
Definition of what the agent decides
Agreed scope for Day 3
Day 3
3 sessions
Map Data Availability and Gaps
Determine what data exists, where it lives, and what needs resolving before any build begins.
Data source inventory
Gap assessment
Readiness rating

Objective of the day

Before any design begins, the team maps what data actually exists, which systems it lives in, and what access looks like in practice. This session surfaces gaps early, before they become blockers in the build phase.

Session 1
System and Data Inventory Walk through the ERP, MES, historian, and operational data sources relevant to the selected decision.
Session 2
Data Access and Quality Check Assess what is accessible, in what format, at what frequency, and where quality is unreliable.
Session 3
Gap Prioritization Rank data gaps by their impact on the decision. Identify which need resolution before build and which can be deferred.

End of Day 3 outputs

Inventory of data sources and systems
Access and quality assessment
Ranked gap list with recommendations
Data readiness rating for the use case
Day 4
3 sessions
Define Decision Logic and Data Product Scope
Define what the agent decides, what data it uses, and the limits it operates within.
Decision logic map
Data product scope
Constraint and approval rules

Objective of the day

The team defines exactly what the AI agent will do: what data it draws on, what logic it applies, what it recommends, and where human review is required. This becomes the design specification for the build phase.

Session 1
Decision Logic Mapping Define the inputs, evaluation logic, thresholds, and outputs of the selected decision end to end.
Session 2
Data Product Definition Specify what data products are needed to feed the decision reliably: what they contain, how often they refresh, and who owns them.
Session 3
Constraint and Approval Design Define operating limits, override conditions, escalation paths, and who approves which recommendations.

End of Day 4 outputs

Decision logic diagram
Data product specification
Constraint and approval rules
Design input for Day 5 validation
Day 5
3 sessions
Validate Scenarios and Define Next Steps
Test the decision logic against real plant scenarios and agree on the first build milestone.
Validated scenarios
Confidence rating
Implementation path

Objective of the day

The decision logic defined on Day 4 is tested against real plant scenarios drawn from operational history. Gaps in the logic are surfaced and resolved. The team agrees on what gets built, in what order, and what a successful first milestone looks like.

Session 1
Scenario Walkthrough Run the decision logic through four to five real scenarios: machine breakdown, material shortage, delayed order, capacity crunch, and shift handover.
Session 2
Gap Resolution and Confidence Rating Capture what the logic handles well and what needs refinement. Assign a confidence rating per scenario.
Session 3
Implementation Planning Define the build sequence, dependencies, team responsibilities, and the first delivery milestone with a clear definition of done.

End of Day 5 outputs

Scenario test results and confidence rating
Refined decision logic
Implementation roadmap
Agreed definition of done for Phase 1
